# GFP HTTP near-neighbour server

`gfp_http_server.py` serves LillyMol GFP near-neighbour searches over HTTP. It is
intended as a replacement for the older ZeroMQ-based GFP server.

The server loads a GFP fingerprint file into memory at startup. Each request
supplies a query SMILES, which is converted to a standard LillyMol GFP
fingerprint and searched against the in-memory fingerprints. Replies are returned
as either binary protobuf or protobuf JSON.

## Prerequisites

There are two different environments to think about: the process running the
server, and any client process talking to the server. Their requirements are not
the same.

### Server environment

The server is a Python process that calls LillyMol C++ code through pybind. The
server environment must therefore contain:

- a Python interpreter compatible with the one used to build the pybind extension;
- the `lillymol_gfp_server` pybind module, usually found via `PYTHONPATH`;
- the generated protobuf Python modules, also usually found via `PYTHONPATH`;
- `fastapi`;
- `uvicorn`;
- `protobuf`.

For running the checked-in HTTP server test, `httpx` is also needed because it is
used by FastAPI's in-process test client.

With an already activated virtual environment, the runtime packages can be
installed with either `pip`:

```shell
python -m pip install fastapi uvicorn protobuf
```

or `uv`:

```shell
uv pip install fastapi uvicorn protobuf
```

For tests:

```shell
uv pip install fastapi uvicorn protobuf httpx
```

The server examples below assume the virtual environment has already been
activated, so `python` refers to the virtual-environment interpreter.

A standard LillyMol build can copy the pybind shared libraries and generated
protobuf modules to stable locations with:

```shell
cd ${LILLYMOL_HOME}/src
./copy_shared_libraries.sh ${LILLYMOL_HOME}/lib
```

The shell wrappers in `${LILLYMOL_HOME}/contrib/bin` use those stable locations
when setting `PYTHONPATH`.

### Client environment

Clients using the JSON endpoints do not need LillyMol, pybind, FastAPI, or
protobuf. They only need an HTTP client. For example, `curl` is sufficient for:

- `POST /search.json`
- `POST /batch.json`
- `GET /healthz`

Clients using the binary protobuf endpoints need code capable of constructing and
parsing the protobuf messages defined in
[`nn_request.proto`](/src/Utilities/GFP_Tools/nn_request.proto). A Python binary
protobuf client needs the generated `nn_request_pb2` module and the Python
`protobuf` package, but still does not need FastAPI, uvicorn, or the LillyMol
pybind module unless it is also running a server. The `gfp_http_client.sh` wrapper
sets `PYTHONPATH` so the checked-in client can find those generated protobuf
modules.

## Input fingerprint file

The GFP file containing the fingerprints to search must contain the standard
fingerprints expected by the server. This is the default generated by
`gfp_make.sh`:

```shell
gfp_make.sh collection.smi > collection.gfp
```

The default `gfp_make.sh` output includes the SMILES for each molecule. Keeping
those SMILES in memory is convenient because neighbour results can include both
identifiers and SMILES, but it consumes server memory and increases response
size.

If you already have another way to retrieve SMILES from identifiers, start the
server with `--no-store-smiles`. In that mode, neighbour SMILES are not stored in
memory and cannot be returned in search results.

For most use cases, retaining SMILES in the server is the more convenient choice.

## Start the server

Build the pybind module first and copy the runtime Python components to their
stable locations:

```shell
cd ${LILLYMOL_HOME}/src
bazel build //pybind:lillymol_gfp_server //Utilities/GFP_Tools:nn_request_py_proto
./copy_shared_libraries.sh ${LILLYMOL_HOME}/lib
```

Then start the server with the wrapper. The wrapper sets `LILLYMOL_HOME` if
needed and prepends `${LILLYMOL_HOME}/src` and `${LILLYMOL_HOME}/lib` to
`PYTHONPATH`.

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_server.sh \
  --gfp pool.gfp \
  --host 127.0.0.1 \
  --port 8000
```

When running directly from a Bazel development tree, the equivalent command is:

```shell
PYTHONPATH=bazel-bin/pybind:bazel-bin \
  python Utilities/GFP_Tools/gfp_http_server.py \
    --gfp pool.gfp \
    --host 127.0.0.1 \
    --port 8000
```

The pool is loaded once at startup. To reload the pool, kill the existing process
and restart it.

## Endpoints

| Endpoint | Request | Response |
|----------|---------|----------|
| `GET /healthz` | none | JSON status and pool size |
| `POST /search` | binary `gfp_server.NnRequest` | binary `gfp_server.NnReply` |
| `POST /search.json` | protobuf JSON `NnRequest` | protobuf JSON `NnReply` |
| `POST /batch` | binary `gfp_server.NnBatchRequest` | binary `gfp_server.NnBatchReply` |
| `POST /batch.json` | protobuf JSON `NnBatchRequest` | protobuf JSON `NnBatchReply` |

The protobuf schema is
[`nn_request.proto`](/src/Utilities/GFP_Tools/nn_request.proto). Neighbour lists
use [`nearneighbours.proto`](/src/Utilities/GFP_Tools/nearneighbours.proto).

## JSON examples

Nearest neighbours by count:

```shell
curl -X POST http://127.0.0.1:8000/search.json \
  -H 'Content-Type: application/json' \
  -d '{"smiles":"CCO","id":"ethanol","nbrs":3}'
```

Neighbours within a maximum distance:

```shell
curl -X POST http://127.0.0.1:8000/search.json \
  -H 'Content-Type: application/json' \
  -d '{"smiles":"CCO","id":"ethanol","distance":0.25}'
```

Batch request:

```shell
curl -X POST http://127.0.0.1:8000/batch.json \
  -H 'Content-Type: application/json' \
  -d '{
        "request": [
          {"smiles":"CCO","id":"ethanol","nbrs":3},
          {"smiles":"c1ccccc1","id":"benzene","nbrs":3}
        ]
      }'
```

## Python client

A simple command-line client is available via the `gfp_http_client.sh` wrapper.
It talks to the binary protobuf `/search` and `/batch` endpoints. The default
output is protobuf text format. Use `--output-format json` for protobuf JSON, or
`--output-format tdt` for LillyMol nearest-neighbour TDT output. The older
`--json` option is retained as an alias for `--output-format json`.

Nearest neighbours by count:

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_client.sh \
  --url http://127.0.0.1:8000 \
  --smiles 'CCO' \
  --id ethanol \
  --nbrs 3
```

Neighbours within a maximum distance:

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_client.sh \
  --url http://127.0.0.1:8000 \
  --smiles 'CCO' \
  --id ethanol \
  --distance 0.25 \
  --json
```

Use `--niter` for repeated lookups when doing simple timing checks.

The client can also read a whitespace-separated SMILES file. Column 1 is the
SMILES and column 2 is the identifier. Extra columns are ignored. By default,
each input record is sent as one `/search` request, and one reply is written for
each input record.

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_client.sh \
  --url http://127.0.0.1:8000 \
  --input queries.smi \
  --nbrs 3
```

For larger query files, use `--batch-size` to send groups of records to the
`/batch` endpoint. The output is still one reply per input molecule, written in
input order.

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_client.sh \
  --url http://127.0.0.1:8000 \
  --input queries.smi \
  --batch-size 100 \
  --nbrs 3
```

Use `--input -` to read the SMILES records from standard input.

For compatibility with existing LillyMol nearest-neighbour tools, the client can
write TDT output. Each needle is written as `$SMI<...>`, `PCN<...>`, zero or more
neighbour `$SMI<...>`, `PCN<...>`, `DIST<...>` groups, and a final `|` record
separator. Molecules with no neighbours are written as just the needle SMILES,
needle identifier, and `|`. Requests that return an error are reported as
warnings on stderr and are not written to stdout.

```shell
${LILLYMOL_HOME}/contrib/bin/gfp_http_client.sh \
  --url http://127.0.0.1:8000 \
  --input queries.smi \
  --batch-size 100 \
  --nbrs 3 \
  --output-format tdt > queries.nn
```

## Binary protobuf examples

For programmatic clients, use the binary protobuf endpoints:

```shell
curl -X POST http://127.0.0.1:8000/search \
  -H 'Content-Type: application/x-protobuf' \
  --data-binary @request.pb \
  --output reply.pb
```

and for batches:

```shell
curl -X POST http://127.0.0.1:8000/batch \
  -H 'Content-Type: application/x-protobuf' \
  --data-binary @batch_request.pb \
  --output batch_reply.pb
```

## Status values

`NnReply.status` is one of:

- `OK`
- `NO_DIRECTIVE`
- `NO_SMILES`
- `BAD_SMILES`
- `SERVER_ERROR`

A request can succeed with `OK` and still return zero neighbours, for example
when a distance cutoff finds no matches.

## Notes

- The service does not implement reload or shutdown endpoints. Restart the
  process to load a new pool.
- The C++ search call releases the Python GIL and is run through FastAPI's
  threadpool.
- For large production deployments, run multiple server processes rather than
  relying on one Python process for all CPU parallelism.
